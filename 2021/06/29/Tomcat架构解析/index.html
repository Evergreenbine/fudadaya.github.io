Tomcat学习笔记

Tomcat简介

Tomcat是一款完全免费开源的Servlet容器实现，其实我更愿意说它是一款支持动态网页的http服务器。它支持的协议有http1，http2以及ajp协议。Tomcat的在8.0版本后开始，开始支持Servlet3.0以上的规范，WebSocket协议的实现；在8.5版本后移除BIO的实现方式，而采用NIO/NIO2(AIO)的方式进行了重新实现。

Tomcat的启动流程概述

在学习Tomcat之前，一直有个问题困惑着我的小脑袋，那就是一请求对应的是一个线程去处理吗？这小问题从我学习mvc框架的时候就产生了，不过所幸的是，在tocmat项目里的serverStartup.txt解决了我的疑惑，对应的时序图文档在这。

下面是我比较关心一些核心的点

  • Catalina(Servlet容器的初始化)的初始化

    • initDirs()：为容器设置一些变量，比如Catalina.home、Catalina.base

    • createStartDigester(): 创建XML解析器以解析Server.xml、Web.xml文件

    • 加载和解析Server.xml

  • Catalina的start

    • 启动配置在Server标签下的Service服务
    • 在容器的的生命周期中，会有一个守护线程去检查容器是否发生了改变，比如说web.xml文件，如果改变则会重新加载

  • Tomcat从http端口监听请求

    1. tomcat维护了一个线程池，线程池中的每个线程监听request请求，每个线程会在ServerSocket.accept()调用后被唤醒
    2. 线程池会分配任务线程去处理请求
    3. 对于http1.1协议，会调用Http11Processor处理器的process方法去解析协议，值得我们注意的是，这个处理器会一直监听Socket中的输入流，直到socket连接断开或者端口被关闭了。
    4. 在请求被解析完成后，一些请求行，请求头等会被封装成一个Coyote Request，这个请求包含了所有Http的信息，像服务器名称、端口号、协议等
    5. 在处理器里有一个适配器 CoyoteAdapter，一旦请求被解析完成，就会调用CoyoteAdapter的Service(),在Service()里，Request请求对象中包含一个CoyoteRequest和CoyoteResponse对象，这两个实例实现了HTTPRequest和HttpResponse接口，适配器解析这些请求信息会和一个map联系起来。
    6. 当解析完成，CoyoteAdapter会调用它的容器(StandardEngine)的invoke(request,response)方法。

Tomcat的总体架构

Tomcat总体架构可以大致可以划分如下图，总体分为了6大部分。

  • Catalina: Servlet容器
  • Coyote: 连接器
  • EL: JavaEL表达式的支持
  • Jasper：Jsp的支持
  • Naming：命名服务的支持，如JNDI
  • Juli：服务器日志

比较值得我们关注的主要是Coyote和Catalina两大部分。

Coyote连接器组成

什么是Coyote？

Coyote是Tomcat连接器框架的名称，是Tomcat服务器提供的供客户端访问的外部接口。客户端通过Coyote与服务器建立连接，发送请求并接受回应。

Coyote的组成？

Coyote连接器主要作用是处理Socket请求和解析Http协议，它主要包含两个重要的部分。

  • AbstractEndpoint<S>：主要用于建立Socket连接，它支持NIO，NIO2（AIO)，Apr（C++实现）的方式实现，Apr库需额外安装相关的包。

  • AbstractProcessor：协议处理接口，用于解析Http请求，Http1.1的实现类为Http11Processor。
  • Adapter：请求适配器，主要作用是将CoyoteRequest包装为ServletRequest。

Catalina（Servlet容器）的组成

Container的组成

容器 描述
Engine 表示整个Catalina的Servlet引擎，用来管理多个虚拟站点，一个Service最多只有一个Engine，一个Engine包含多个Host。
Host 代表一个站点，给Tomcat配置多个Host，可以用一个Tomcat部署多个域名的站点。
Context 表示一个Web应用程序，就是一个项目啦。
Wrapper 表示一个基本Servlet，也即是一个Servlet。
